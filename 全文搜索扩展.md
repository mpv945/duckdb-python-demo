DuckDB 的 `fts` 是**全文搜索扩展**,用来在大段文本里快速做"关键词搜索 + 相关度排序",类似 SQLite 的 FTS5 或者一个迷你版 Elasticsearch,但完全用 SQL 实现,不需要额外的搜索服务。

## 它解决什么问题

普通 SQL 的 `LIKE '%关键词%'` 只能做简单字符串匹配,没法按"相关度"排序,而且性能在大表上很差(全表扫描)。SQL 没有提供很好的方式来表达"给我所有关于野鸭的文档"这类问题,LIKE 模式匹配能力有限。FTS 扩展通过**倒排索引(Inverted Index)**解决这个问题:倒排索引建立关键词到包含该关键词的文档 ID 的映射,先识别查询字符串里的关键词,匹配倒排索引,从而快速定位出包含这些关键词的文档,然后用 Okapi BM25 评分函数(经典的相关度算法,综合考虑关键词出现次数、文档长度、关键词在整个语料库中的稀有程度)给结果排序。

特性总结:
- BM25 相关度排序——文本相关性评分的黄金标准
- Porter 词干提取——"running" 变成 "run","failed" 变成 "fail",这样搜 "run" 也能匹配到含 "running" 的文档
- 自动跳过停用词,比如 "the"、"a"、"is" 这些没有信息量的常见词
- 支持多种语言的词干提取器
- 重音符号归一化处理

## 适用场景

日志检索、文档/工单内容搜索、客服工单分类查找、文章/评论关键词检索等"文本量大、需要按相关度找内容"的场景。不需要额外的服务,不需要额外的基础设施,只需要三条 SQL 语句 —— 这是它最大的优势:不用额外起一个 Elasticsearch 集群。

## 怎么用:三步走

**1. 准备数据表**

```python
import duckdb

conn = duckdb.connect()

conn.execute("""
    CREATE TABLE articles (
        id INTEGER,
        title VARCHAR,
        body VARCHAR
    )
""")
conn.execute("""
    INSERT INTO articles VALUES
    (1, 'Database Performance Tips', '如何优化你的 SQL 查询以获得更好的性能...'),
    (2, 'Login Security Best Practices', '通过合理的身份认证防止未授权访问...'),
    (3, 'Query Optimization Guide', '编写高效数据库查询的技巧...')
""")
```

**2. 建索引**

PRAGMA create_fts_index('documents', 'document_identifier', 'text_content', 'author') 这种语法:第一个参数是表名,第二个是唯一标识列(文档 ID),后面是要建索引的文本列(可以指定多个,也可以用 `*` 表示所有 VARCHAR 列):

```python
conn.execute("PRAGMA create_fts_index('articles', 'id', 'title', 'body')")
```

**3. 查询并按相关度排序**

```python
df = conn.execute("""
    SELECT id, title, score
    FROM (
        SELECT *, fts_main_articles.match_bm25(id, 'query optimize performance') AS score
        FROM articles
    ) sq
    WHERE score IS NOT NULL
    ORDER BY score DESC
""").df()
print(df)
```

`match_bm25(文档ID列, 查询字符串)` 返回相关度得分,**没匹配到的行返回 NULL**,所以一定要加 `WHERE score IS NOT NULL` 过滤掉无关结果。

## 进阶用法

**只搜某个字段(比如只搜 title,不搜 body):**
```sql
SELECT *, fts_main_articles.match_bm25(id, '关键词', fields := 'title') AS score
FROM articles
```

**短语搜索(要求多个词紧邻出现):**
```sql
WHERE match_fts('articles', '"login security"') IS NOT NULL
```

**结合普通字段过滤(混合查询,FTS 最大的优势之一):**
```sql
SELECT id, title, score
FROM (
    SELECT *, fts_main_articles.match_bm25(id, 'database') AS score
    FROM articles
) sq
WHERE score IS NOT NULL AND length(body) > 1000
ORDER BY score DESC
```

**指定词干提取语言(中文文本一般不需要,英文/德文等有词形变化的语言才需要):**
```sql
-- 默认 porter(英文),也支持 german / dutch / french / spanish 等
PRAGMA create_fts_index('articles', 'id', 'title', 'body', stemmer = 'german');
```

## 几个重要的注意事项

1. FTS 索引不会随着原表数据变化自动更新。如果表数据变了(增删改),需要手动重建索引:
   ```python
   conn.execute("PRAGMA create_fts_index('articles', 'id', 'title', 'body', overwrite = 1)")
   ```
   加 `overwrite = 1` 才能覆盖已有索引,否则会报错说索引已存在。

2. **索引建在哪儿**:索引会建在一个新创建的 schema 下,schema 名字根据表名生成,比如表是 main.articles,schema 就叫 fts_main_articles。

3. 索引建好之后,原始输入表理论上可以删掉,FTS 的检索宏依然能用(只是返回结果会是 NULL,因为没有原表数据可关联) —— 实际使用中通常还是保留原表,因为你需要原表的其他列(标题、正文等)。

4. **中文分词的限制**:目前 FTS 扩展主要面向**空格分词的语言**(英文等),中文这种没有天然空格分隔的语言,直接用默认配置效果会打折扣(它按非字母字符切词,中文会被切得很碎或者整句当一个 token)。如果要做中文全文检索,实际项目中通常需要先用分词工具(比如 jieba)把中文文本预处理成空格分隔的词,再喂给 FTS 索引,或者考虑专门的中文搜索引擎方案。

5. **和普通索引的区别**:这不是 B-tree 那种加速等值查询的索引,是专门为"模糊关键词检索+排序"设计的,定位完全不同,不要拿它替代普通的主键/外键索引。
