DuckDB 可以同时 `ATTACH` MySQL 和 PostgreSQL,把两者当成"普通表"一样跨库查询和写入,中间完全不需要落地文件,DuckDB 本身就是数据搬运的桥梁。

## 整体思路

1. 用 `mysql` 扩展 ATTACH 到 MySQL(只读)
2. 用 `postgres` 扩展 ATTACH 到 PostgreSQL(读写)
3. 用 `CREATE TABLE ... AS SELECT` 或 `INSERT INTO ... SELECT` 直接把 MySQL 的数据写进 PostgreSQL

## 完整代码

```python
import duckdb

conn = duckdb.connect()

# 1. 安装并加载两个扩展(一般会自动加载,显式写更稳妥)
conn.execute("INSTALL mysql; LOAD mysql;")
conn.execute("INSTALL postgres; LOAD postgres;")

# 2. ATTACH MySQL 源库(连接串是 PostgreSQL 风格的 key=value,不是 MySQL 自己的格式)
conn.execute("""
    ATTACH 'host=mysql_host port=3306 user=myuser password=mypass database=mydb'
    AS mysql_db (TYPE mysql, READ_ONLY)
""")

# 3. ATTACH 目标 PostgreSQL 库(libpq 风格连接串)
conn.execute("""
    ATTACH 'host=pg_host port=5432 user=pguser password=pgpass dbname=pgdb'
    AS pg_db (TYPE postgres)
""")

# 4. 整表搬运:在 PostgreSQL 里新建表并写入 MySQL 的全部数据
conn.execute("""
    CREATE TABLE pg_db.target_table AS
    SELECT * FROM mysql_db.source_table
""")
```

ATTACH 用的连接串是 PostgreSQL 风格的连接字符串(不是 MySQL 自己的连接字符串!),是一组 key=value 格式的参数,这点容易搞混要注意。

## 如果目标表已存在,只追加数据

```python
conn.execute("""
    INSERT INTO pg_db.target_table
    SELECT * FROM mysql_db.source_table
""")
```

## 边迁移边转换/过滤(常见场景)

不一定要整表照搬,可以加 WHERE、JOIN、类型转换等:

```python
conn.execute("""
    INSERT INTO pg_db.target_table (id, name, created_at)
    SELECT id, name, created_at
    FROM mysql_db.source_table
    WHERE created_at >= '2026-01-01'
""")
```

## 大表增量迁移(分批,避免一次性占满内存/锁表)

对于较大的表,可以按时间窗口或主键范围分批 INSERT,重复执行直到迁移完成,比如按月份分片:

```python
months = ["2026-01", "2026-02", "2026-03"]  # 按需生成

for month in months:
    conn.execute(f"""
        INSERT INTO pg_db.target_table
        SELECT * FROM mysql_db.source_table
        WHERE date_trunc('month', created_at) = DATE '{month}-01'
    """)
    print(f"已迁移 {month} 的数据")
```

## 先建好 schema 再迁移数据(更稳妥的做法)

如果 MySQL 和 PostgreSQL 的字段类型有差异(比如 MySQL 的 `TINYINT(1)` 对应 PostgreSQL 的 `BOOLEAN`),建议先手动定义目标表结构,再用 `INSERT INTO ... SELECT` 写入,而不是依赖 `CREATE TABLE AS` 自动推断类型:

```python
conn.execute("""
    CREATE TABLE pg_db.target_table (
        id INTEGER PRIMARY KEY,
        name VARCHAR,
        is_active BOOLEAN,
        created_at TIMESTAMP
    )
""")

conn.execute("""
    INSERT INTO pg_db.target_table
    SELECT id, name, is_active::BOOLEAN, created_at
    FROM mysql_db.source_table
""")
```

## 性能调优建议

如果表很大,可以调大 threads 和 memory_limit,并通过 pg_connection_limit 和 pg_pages_per_task 控制对源/目标数据库的并发压力,数值太高反而会给生产库带来连接压力:

```python
conn.execute("SET threads = 8")
conn.execute("SET memory_limit = '8GB'")
```

## 用环境变量管理密码(避免硬编码)

```python
import os

mysql_conn_str = f"host={os.environ['MYSQL_HOST']} port=3306 user={os.environ['MYSQL_USER']} password={os.environ['MYSQL_PASSWORD']} database=mydb"
pg_conn_str = f"host={os.environ['PG_HOST']} port=5432 user={os.environ['PG_USER']} password={os.environ['PG_PASSWORD']} dbname=pgdb"

conn.execute(f"ATTACH '{mysql_conn_str}' AS mysql_db (TYPE mysql, READ_ONLY)")
conn.execute(f"ATTACH '{pg_conn_str}' AS pg_db (TYPE postgres)")
```

## 几个注意点

1. **加 READ_ONLY 给源库**:MySQL 端建议加 `READ_ONLY`,避免误操作改了源数据。
2. **schema 不一致时**:DuckDB 不会自动帮你处理 MySQL → PostgreSQL 间所有类型差异,复杂表建议先 `DESCRIBE mysql_db.source_table` 看看字段类型,再决定要不要手动建表。
3. **事务边界**:跨多个已 attach 的数据库执行事务时,每个数据库会各自开自己的事务,读取没问题,但写入跨库事务会更复杂,大批量迁移建议分批提交,不要指望一个大事务里完成所有跨库写入。
4. **如果两边表结构差异很大**:也可以走中间落地 Parquet 的方式(MySQL → Parquet → PostgreSQL),适合需要审计、重试、或者迁移过程中需要人工检查中间结果的场景。
