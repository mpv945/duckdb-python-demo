## 第三步：一步一步创建 DuckDB 分析项目## 1. 创建并激活 Conda 环境
创建一个独立的 Python 环境（DuckDB 官方要求 Python 3.9+，此处推荐使用 3.10）：

conda create -n duckdb_env python=3.10 -y
conda activate duckdb_env

## 2. 安装项目依赖与打包工具
由于后续需要离线移植，我们必须在当前激活的 Conda 环境中安装 conda-pack 核心工具。同时使用 uv 快速安装 duckdb 及其常用的数据分析辅助库（如 pandas 和 pyarrow）：

# 安装离线打包工具
conda install -c conda-forge conda-pack -y
# 使用 uv 加速安装 DuckDB 及其数据栈
uv pip install duckdb pandas pyarrow

## 3. 编写 DuckDB 分析脚本
在项目目录中创建一个名为 analysis.py 的脚本。该脚本将创建一个持久化数据库，并演示如何加载内存数据及导出分析结果：

import duckdbimport pandas as pd
def main():
    print("--- 正在初始化 DuckDB 持久化数据库 ---")
    # 1. 创建或连接到本地持久化数据库文件
    con = duckdb.connect("offline_analytics.db") #
    
    # 2. 准备一份测试数据（模拟 Pandas DataFrame 导入）
    df = pd.DataFrame({
        "user_id":,
        "sales": [250.5, 550.0, 120.0],
        "region": ["North", "South", "North"]
    })
    
    # 3. 将 Pandas DataFrame 注册并写入 DuckDB 本地表
    con.sql("CREATE TABLE IF NOT EXISTS sales_data AS SELECT * FROM df")
    
    # 4. 执行 SQL 聚合分析
    print("\n--- 执行 SQL 聚合分析结果 ---")
    result_rel = con.sql("""
        SELECT region, SUM(sales) AS total_sales, COUNT(user_id) AS user_count
        FROM sales_data
        GROUP BY region
    """)
    result_rel.show() #
    
    # 5. 将结果转换回 Pandas DataFrame 并导出为本地 CSV
    analysis_df = result_rel.df() #
    analysis_df.to_csv("summary_report.csv", index=False)
    print("\n[成功] 报告已导出至 'summary_report.csv'")
    
    # 6. 关闭数据库连接
    con.close() #
if __name__ == "__main__":
    main()

**如果内网机器完全没有 Conda（也没有 Miniconda），推荐方案如下：**

### 最佳实践：同时打包/传输 Miniconda 安装程序 + conda-pack 环境

#### 1. 在**开发机（有网）**准备两样东西

```bash
# 第一步：打包你的项目环境（同之前）
conda activate myproject
conda install -c conda-forge conda-pack
conda pack -n myproject -o myproject.tar.gz --ignore-editable-packages
```

**第二步：下载 Miniconda 安装程序**（推荐 Miniconda，比完整 Anaconda 小得多）

- 去官网下载对应平台的 Miniconda（Linux / Windows / macOS）：https://docs.anaconda.com/free/miniconda/
- 或者直接用命令下载（Linux 示例）：
  ```bash
  wget https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-x86_64.sh
  ```

把下面**两个文件**一起传输到内网机器：
- `Miniconda3-latest-Linux-x86_64.sh`（或对应平台安装包）
- `myproject.tar.gz`

#### 2. 在**内网目标机**上安装和还原

```bash
# 1. 安装 Miniconda（离线安装）
chmod +x Miniconda3-latest-Linux-x86_64.sh
./Miniconda3-latest-Linux-x86_64.sh -b -p ~/miniconda3   # -b 表示静默安装，-p 指定路径

# 把 Miniconda 加到 PATH（推荐加入 ~/.bashrc）
export PATH="~/miniconda3/bin:$PATH"
source ~/.bashrc

# 2. 解压打包的环境
mkdir -p ~/miniconda3/envs/myproject
tar -xzf myproject.tar.gz -C ~/miniconda3/envs/myproject

# 3. 激活并修复环境（关键步骤）
conda activate myproject
conda-unpack

# 4. 验证并运行
python --version
python -c "import numpy; print('OK')"

cd /path/to/your/project
python main.py
```

**安装后激活环境的方式**（以后每次用）：
```bash
source ~/miniconda3/bin/activate myproject
# 或
conda activate myproject   # 如果 PATH 已配置好
```

### 更极致的“完全独立”方案（不依赖 conda 命令）

conda-pack 打包后的环境本质上就是一个完整的 Python + site-packages 目录，你可以**直接用里面的 python 可执行文件**运行，而不走 conda activate。

```bash
# 解压到任意目录（不需要 conda）
mkdir -p /opt/myproject-env
tar -xzf myproject.tar.gz -C /opt/myproject-env

# 直接使用
/opt/myproject-env/bin/python /path/to/your/project/main.py
```

或者写一个启动脚本 `run.sh`：
```bash
#!/bin/bash
export PATH="/opt/myproject-env/bin:$PATH"
export LD_LIBRARY_PATH="/opt/myproject-env/lib:$LD_LIBRARY_PATH"  # Linux 需要
python main.py
```

这种方式**完全不需要安装 Miniconda**，适合极致轻量或严格受限的环境。

### 其他注意事项

- **平台必须完全一致**：操作系统、架构（x86_64 / aarch64 等）要一样，否则二进制不兼容。
- **文件大小**：Miniconda 安装包 ~100MB 左右，环境包可能几个 GB，提前准备好传输方式。
- **权限问题**：建议用普通用户安装，避免 root。
- **pip 包兼容性**：大多数可以，但极少数编译型包可能需要额外处理。
- **Micromamba** 替代方案：如果不想用完整 Miniconda，可以考虑 Micromamba（更小、更快），但操作稍复杂一些。

**推荐优先级**：
1. Miniconda + conda-pack（最稳妥，功能完整）
2. 纯解压 + 直接用 `bin/python`（最轻量）

**内网完全无 Conda 时（直接运行“绿色版”）的最佳实战流程**

以下结合微信文章《基于conda实现Python环境离线部署》和我在生产环境中的实际经验（多个项目跨机房/内网部署），给你一个**完整、可落地的一步步操作指南**。重点针对**内网无任何 Conda** 的场景，实现“复制即运行”。

### 前提条件（必须满足）
- **开发机（有网）** 和 **内网目标机** 的**操作系统 + 架构完全一致**（例如都是 CentOS 7.9 x86_64 或 Ubuntu 20.04 x86_64）。这是成败关键，否则二进制不兼容。
- 项目代码准备好（推荐放在独立目录）。
- 传输介质：U盘、内网共享盘、scp 等。

---

### 一、在开发机（有网）打包

1. **创建干净环境并安装依赖**  
   ```bash
   conda create -n myproject python=3.10 -y
   conda activate myproject

   # 优先用 conda-forge 安装（更稳定）
   conda install -c conda-forge numpy pandas scipy scikit-learn etc. -y

   # 再用 pip 安装剩余依赖
   pip install -r requirements.txt
   ```

2. **安装打包工具**  
   ```bash
   conda install -c conda-forge conda-pack -y
   ```

3. **打包环境**（推荐带忽略可编辑包）  
   ```bash
   # 打包当前激活的环境
   conda pack -n myproject -o myproject.tar.gz --ignore-editable-packages

   # 如果环境路径是明确的，也可以用 -p
   # conda pack -p /path/to/envs/myproject -o myproject.tar.gz
   ```

   打包完成后得到 `myproject.tar.gz`（大小通常几百MB 到 几GB）。

---

### 二、内网目标机操作（无 Conda）

#### 推荐方案：**纯绿色版直接运行**（最轻量，无需安装 Miniconda）

1. **传输文件**  
   把 `myproject.tar.gz` 和你的**项目代码目录** 复制到内网机器（例如 `/opt/myproject/`）。

2. **解压环境**  
   ```bash
   # 创建目录（推荐独立路径，不要随意改动）
   mkdir -p /opt/myproject-env
   tar -xzf /path/to/myproject.tar.gz -C /opt/myproject-env
   ```

3. **修复环境（关键一步！）**  
   ```bash
   # 进入解压后的 bin 目录执行
   cd /opt/myproject-env/bin
   ./conda-unpack
   ```

   `conda-unpack` 会修复所有硬编码路径，使环境可移植。

4. **直接运行项目**（无需 conda activate）  
   有两种常用方式：

**DuckDB 扩展离线安装（外网下载 + 内网手动导入）完整实战指南**

DuckDB 扩展文件是 `.duckdb_extension`（或压缩的 `.duckdb_extension.gz`），官方仓库地址为 `https://extensions.duckdb.org`（推荐 https）。内网无网环境下，只能**手动下载后传输并本地安装**。

### 1. 外网机器（有网）准备

#### 步骤：
1. **确认 DuckDB 版本和平台**（非常重要，必须匹配！）
   - 在 DuckDB 中运行：
     ```sql
     SELECT version();
     SELECT current_setting('extension_directory');  -- 查看平台路径
     ```
   - 或命令行：`duckdb --version`

   常见平台名称示例：
   - Linux x86_64 → `linux_amd64`
   - Linux ARM64 → `linux_arm64`
   - Windows x86_64 → `windows_amd64`
   - macOS Intel → `osx_amd64`
   - macOS Apple Silicon → `osx_arm64`

2. **下载扩展文件**
   使用浏览器或命令行下载对应版本的 `.duckdb_extension.gz` 文件。

   **下载 URL 格式**（推荐 https）：
   ```
   https://extensions.duckdb.org/v{版本号}/{平台}/{扩展名}.duckdb_extension.gz
   ```

   示例（v1.5.3 + linux_amd64）：
   - `https://extensions.duckdb.org/v1.5.3/linux_amd64/httpfs.duckdb_extension.gz`
   - `https://extensions.duckdb.org/v1.5.3/linux_amd64/spatial.duckdb_extension.gz`
   - `https://extensions.duckdb.org/v1.5.3/linux_amd64/parquet.duckdb_extension.gz`（很多核心扩展已内置，但仍可手动）

   **批量下载脚本示例**（Linux/macOS，外网机器）：
   ```bash
   VERSION="1.5.3"
   PLATFORM="linux_amd64"
   EXTENSIONS="httpfs spatial parquet excel aws json"

   mkdir -p duckdb-extensions
   for ext in $EXTENSIONS; do
       wget "https://extensions.duckdb.org/v${VERSION}/${PLATFORM}/${ext}.duckdb_extension.gz" -O "duckdb-extensions/${ext}.duckdb_extension.gz"
   done
   ```

3. **解压 gz 文件**（重要！）
   ```bash
   cd duckdb-extensions
   gunzip *.gz   # 得到 .duckdb_extension 文件
   ```

   把整个 `duckdb-extensions` 文件夹复制到 U 盘 / 内网传输介质。

### 2. 内网机器（离线）手动安装和使用

#### 方式一：推荐 —— 使用 `INSTALL` + 本地路径（最简单）
```sql

-- 加载（每次启动 DuckDB 会话都需要）
```

   ```sql
   SELECT current_setting('extension_directory');
   ```

2. 把 `.duckdb_extension` 文件复制到对应子目录，例如：
   ```
   ~/.duckdb/extensions/v1.5.3/linux_amd64/
   ```

3. 然后直接：
   ```sql
   LOAD httpfs;
   ```

#### 方式三：Python / 其他客户端 API
```python

# 安装本地文件
con.execute("INSTALL '/absolute/path/to/httpfs.duckdb_extension'")
```

- **版本严格匹配**：DuckDB 主版本 + 平台必须一致，否则加载失败。

**DuckDB 可以非常优雅地同时连接 MySQL 和 PostgreSQL（简称 Postgres），然后直接进行跨库 JOIN 关联分析**，无需先把数据全部导入 DuckDB。这得益于 DuckDB 的 `mysql` 和 `postgres` 扩展（也叫 scanner）。

### 1. 安装和加载扩展

在 DuckDB 中执行（一次即可，之后会自动加载）：

```sql
INSTALL mysql;
INSTALL postgres;

LOAD mysql;
LOAD postgres;
```

- 如果是离线环境，先在外网用 `INSTALL mysql FROM core;` 和 `INSTALL postgres FROM core;` 下载扩展（`.duckdb_extension` 文件），然后内网手动安装（参考之前的扩展离线方法）。

### 2. 附加（ATTACH）两个数据库

```sql
-- 附加 PostgreSQL
ATTACH 'host=pg_host.example.com port=5432 dbname=your_pg_db user=pg_user password=pg_pass' 
AS pg_db (TYPE postgres, READ_ONLY);   -- 推荐 READ_ONLY 用于分析

-- 附加 MySQL
ATTACH 'host=mysql_host.example.com port=3306 dbname=your_mysql_db user=mysql_user password=mysql_pass' 
AS mysql_db (TYPE mysql, READ_ONLY);
```

**常用连接字符串简化写法**（推荐）：

- Postgres：`'dbname=prod user=analytics host=127.0.0.1 password=xxx' AS pg (TYPE postgres)`
- MySQL：`'host=localhost user=root database=sakila password=xxx' AS mysql (TYPE mysql)`

**提示**：
- 使用 `READ_ONLY` 更安全，尤其生产环境。
- 密码敏感，推荐使用环境变量或 DuckDB Secrets 管理（较新版本支持）。
- 可附加多个同类型数据库，用不同别名。

### 3. 跨库 JOIN 关联查询示例

```sql
-- 示例：MySQL 的订单表 JOIN Postgres 的用户表进行关联分析
SELECT 
    m.order_id,
    m.amount,
    m.order_date,
    p.user_name,
    p.email,
    p.registration_date
FROM mysql_db.orders m
JOIN pg_db.users p 
    ON m.user_id = p.user_id
WHERE m.order_date >= '2025-01-01'
ORDER BY m.amount DESC
LIMIT 1000;
```

**更复杂的关联分析**（聚合 + 多表 JOIN）：

```sql
SELECT 
    p.department,
    COUNT(DISTINCT m.order_id) AS order_count,
    SUM(m.amount) AS total_amount,
    AVG(m.amount) AS avg_amount,
    COUNT(DISTINCT c.customer_id) AS unique_customers
FROM mysql_db.orders m
JOIN pg_db.customers c ON m.customer_id = c.id
JOIN pg_db.departments p ON c.dept_id = p.dept_id
WHERE m.status = 'completed'
GROUP BY p.department
HAVING SUM(m.amount) > 100000
ORDER BY total_amount DESC;
```

### 4. 最佳实践 & 性能优化

- **性能**：DuckDB 会尽量把过滤、投影推送到源数据库（pushdown），速度通常比传统 ETL 快很多。复杂分析（如大 JOIN、聚合）在 DuckDB 内存/向量化引擎中执行。
- **大表建议**：
  - 先在源库创建合适索引（尤其是 JOIN 列）。
  - 用 `WHERE` 条件尽量过滤数据后再 JOIN。
  - 必要时 `CREATE TABLE duckdb_table AS SELECT ...` 把中间结果物化到 DuckDB（.duckdb 文件）。
- **查看已附加数据库**：
  ```sql
  SHOW DATABASES;
  SELECT * FROM duckdb_databases();
  ```
- **断开连接**（可选）：
  ```sql
  DETACH pg_db;
  DETACH mysql_db;
  ```

### 5. 注意事项

- **网络连通性**：DuckDB 机器必须能访问 MySQL 和 Postgres 的端口（3306 / 5432）。
- **版本兼容**：DuckDB 版本与扩展要匹配（推荐用最新稳定版）。
- **认证**：MySQL/Postgres 的用户权限至少需要 SELECT。

**DuckDB 跨 MySQL 和 PostgreSQL 联邦查询的 Pushdown 性能优化指南**

**Pushdown**（下推）是 DuckDB 联邦查询的核心优化机制：它会尽量把**过滤（Filter）**、**投影（Projection，只选需要的列）**、部分**聚合** 和 **JOIN** 下推到 MySQL/PostgreSQL 源数据库执行，从而大幅减少网络传输的数据量、提升整体性能。

以下是生产环境中的**实用优化步骤**（结合官方文档和实战经验）：

### 1. 基础配置优化（强烈推荐）

```sql
-- 附加数据库时开启 pushdown 支持
ATTACH 'host=... dbname=... user=... password=...' 
AS pg_db (TYPE postgres, READ_ONLY=true);

ATTACH 'host=... dbname=... user=... password=...' 
AS mysql_db (TYPE mysql, READ_ONLY=true);

-- Postgres 特有设置（提升 filter pushdown）
SET pg_experimental_filter_pushdown = true;
-- 或在 ATTACH 时启用
-- CALL postgres_attach('...', filter_pushdown=true);
```

- 使用 `READ_ONLY` 模式更安全且利于优化。
- 确保 DuckDB 和扩展版本是最新的（`INSTALL mysql; INSTALL postgres;` 后更新）。

### 2. SQL 编写技巧（让 Pushdown 更有效）

1. **尽早过滤（Early Filter）**  
   把强选择性的 `WHERE` 条件写在最前面，且尽量作用于单个表：
   ```sql
   -- 好：过滤下推到源库
   SELECT ... 
   FROM mysql_db.orders m
   JOIN pg_db.users p ON m.user_id = p.user_id
   WHERE m.order_date >= '2025-01-01'          -- 优先下推
     AND p.status = 'active';
   ```

2. **只选择需要的列（Projection Pushdown）**  
   避免 `SELECT *`，明确列出需要的字段，能显著减少传输数据量。

3. **JOIN 顺序与条件**  
   - 把小表或过滤后数据量小的表放在 JOIN 的驱动侧。
   - JOIN 条件尽量使用等值连接（`=`），便于索引和 pushdown。
   - 复杂 JOIN 时，可用 CTE/Subquery 手动控制下推行为。

4. **避免阻碍 Pushdown 的操作**  
   - 避免在 WHERE 中使用非下推函数（如某些自定义函数）。
   - `WHERE col IN (大量值)` 有时 pushdown 不佳，可改用临时表或 JOIN。
   - 复杂表达式或窗口函数可能无法完全下推，需物化中间结果。

### 3. 源数据库端准备（关键！）

- **MySQL / PostgreSQL 创建合适索引**：
  - JOIN 列、WHERE 过滤列、ORDER BY/GROUP BY 列都需要索引。
  - 复合索引优先覆盖最常用的过滤组合。

- **统计信息更新**：
  ```sql
  -- PostgreSQL
  ANALYZE your_table;
  
  -- MySQL
  ANALYZE TABLE your_table;
  ```

- **权限**：附加的用户至少需要 `SELECT` 权限，必要时授予更多以支持优化。

### 4. 监控与诊断 Pushdown 是否生效

```sql
-- 查看 DuckDB 执行计划（最重要！）
EXPLAIN SELECT ... FROM ...;

-- 或更详细
EXPLAIN ANALYZE SELECT ...;
```

在计划中查找：
- `FILTER`、`PROJECTION` 是否下推到 `POSTGRES_SCAN` 或 `MYSQL_SCAN`。
- `pg_debug_show_queries = true`（临时设置），查看 DuckDB 实际发给 Postgres/MySQL 的 SQL。

### 5. 高级优化技巧

- **物化中间结果**（数据量大时）：
  ```sql
  CREATE TABLE temp_filtered AS 
  SELECT * FROM mysql_db.orders WHERE order_date >= '2025-01-01';
  
  -- 再用 temp_filtered JOIN 另一张表
  ```

- **连接池与并行**：
  ```sql
  SET pg_connection_limit = 16;   -- 根据服务器资源调整
  ```

- **二进制传输**（Postgres）：
  确保 `pg_use_binary_copy = true;`（默认开启），传输更快。

- **分区/分表**：如果源表已分区，DuckDB 能更好地利用分区裁剪（Partition Pruning）。

### 6. 常见性能问题与解决

- **Pushdown 不生效** → 检查 EXPLAIN，确认过滤列有索引；尝试重写 SQL。
- **网络瓶颈** → 优先下推强过滤；必要时把数据拉到 DuckDB 后本地计算。
- **MySQL pushdown 支持相对较弱** → 部分复杂谓词可能无法下推，优先在 Postgres 侧做更多过滤。
- **大结果集** → 使用 `LIMIT` + 分页，或导出到 Parquet 后分析。

**总结最佳实践顺序**：
1. 源库建好索引 + 更新统计信息。
2. SQL 中尽早、明确过滤和投影。
3. 用 `EXPLAIN` 验证 pushdown。
4. 必要时物化中间表。
5. 监控网络流量和查询耗时。

按照以上方式优化后，跨库 JOIN 的性能通常能提升 **数倍到数十倍**（取决于数据量和过滤选择性）。

```sql
    m.order_id,
    m.amount,
    m.order_date,
    p.user_name,
    p.email,
    p.department,
    c.customer_type,
    f.parquet_extra_col,           -- 来自 Parquet 文件

JOIN pg_db.users p 
    ON m.user_id = p.user_id
LEFT JOIN pg_db.customers c 
    ON m.customer_id = c.id
JOIN read_parquet('/path/to/external/data/*.parquet') f 
    ON m.order_id = f.order_id
JOIN read_csv_auto('/path/to/csv/files/*.csv', header=true, types={'date_col': 'DATE'}) csv_data
    ON m.user_id = csv_data.user_id
WHERE m.order_date >= '2025-01-01'
  AND p.status = 'active'
GROUP BY 
    m.order_id, m.amount, m.order_date, p.user_name, 
    p.email, p.department, c.customer_type, 
    f.parquet_extra_col, csv_data.some_value;
```

```python
    """动态判断：已安装则 LOAD，未安装则 INSTALL"""
    try:
        # 查询是否已安装

            SELECT installed 
            FROM duckdb_extensions() 
            WHERE extension_name = '{ext_name}'

        else:
            print(f"🔧 扩展 {ext_name} 未安装，正在安装...")
            con.execute(f"INSTALL {ext_name};")
            con.execute(f"LOAD {ext_name};")
            print(f"✅ 扩展 {ext_name} 安装并加载完成")
            
    except Exception as e:
        print(f"❌ 处理扩展 {ext_name} 时出错: {e}")
        raise

        install_or_load_extension(con, "postgres")

        query = f"""
```
